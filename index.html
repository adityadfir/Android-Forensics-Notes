Android Forensics
What is the approach for Mobile Forensics?
Investigation Preparation
Phase begins when request for examination is received
Involves preparing all the paper work and forms required to document chain of custody, ownership information, device model, purpose, information requestor is seeking
Chain of custody is documentation or paper trail, showing the seizure, custody, control, transfer, analysis, and disposition of physical or electronic evidence
Seizure and Isolation
Evidence usually transported using anti-static bags
Designed to protect electronic components against damage produced by static electricity
During time of seizure, if chance phone is recovered unlocked, disable the passcode
If device is unlocked, try to change USB Debugging, Screen Lock time
Put the device in Airplane mode, disable Wi-Fi, Bluetooth or any radio transmission
Use Farady Bag or RF Isolation Box
Acquisition
Refers to extraction of data from device
Extraction method is decided depending on operating system, make and model
Types of acquisition methods
Manual acquisition
Examiner uses the mobile user interface to browse and investigate
No special tools or technique required here
Limitation is files and data visible through other methods can be verified using this
Changes should be documented as thoroughly as possible
Logical acquisition
Extracting files that are present on logical store such as file system partition
Involves obtaining data types such as text messages, call history, and pictures from phone
Works on concept of Application Programming Interface (API) for synchronizing phones content with computer
Types of evidence collected
Call Logs
SMS

Filesystem acquisition
Logical procedure refers to extraction of full file system from mobile device
Sometimes help in recovering contents that are deleted from device
Physical acquisition
Making bit-by-bit copy of entire flash storage device, equivalent to full image of hard drive
Data extracted using this method is usually in form of raw data that can then be further parsed to obtain file system information or human-readable data
Ensures that original evidence is not altered
Examination and Analysis
Use different software and tools to extract data from memory image
There is no single tool that can be used in all cases
Reporting
Documentation should be done throughout the process, noting down what was done in each phase
Data extracted from mobile device should be clearly presented to recipient so it can be imported into other software for further analysis
Android Architecture
OS assumes the responsibility for managing the resources of system and provide way for applications to talk to hardware or physical components in order to accomplish certain tasks
Powers mobile phones, manages memory and processes, enforces security, network troubleshooting
Android is open-source making it access freely, modifying it and use software according to requirements of manufacturers
Layers of Android Architecture
Linux Kernel
Android built on top of Linux kernel
Positioned at bottom of software stack and provides level of abstraction between device hardware and upper layers
Kernel contains drivers that can facilitate this process
All core functionalities of Android, such process management, memory management, security and networking managed by Linux kernel
User-based permission model, Process isolation, Extensible mechanism for secure IPC
Hardware Abstraction Level
Allows higher level, Java API framework, to work with mobile device hardware with standard interfaces
Provide interfaces for different types of hardware components, like Bluetooth or camera
Android Runtime
Each application runs in its own process with its own instance of ART
Run multiple virtual machines on low-memory devices by executing DEX
Native C/C++ Libraries
core android system components and services, HAL and ART, are built from native code
Java API Framework
Allows developer to create applications using modular system components and services as building blocks
View system allows to build application user interface and include lists, grids, text boxes, buttons and so on
Resource Manager provides access to non-code components of application
Notification manager allow applications to display custom alerts
Activity manager manages the lifecycle as application, and their back stack
Content provider allows applications to access other applications data and share
Application Layer
Consists of applications which are programs that users directly interact with
Application Sandboxing
To isolate applications from each other, Android takes advantage of Linux-based protection model
Each user assigned as unique User ID(UID)
Users are segregated so one user does not have access to data of another
If application tries to do something malicious, it can do it only within the context and with permissions it has
Sandboxing done at kernel level
Security between applications and systems at process level is ensured through Linux facilities like user and group IDs
Application Signing
All apps need to be digitally signed with certificate before they can be installed on device
Main purpose is to identify the author of the app
Apps are often self-signed
Developers hold certificate private key
To distribute the app, it needs to be signed with own certificate
Android Boot process
Boot ROM code execution
Execution starts with ROM code execution
Code is specific to device
Consists of 2 steps
Step A
When ROM code executed, initialized device hardware and tries to detect boot media
Boot ROM code scans until finds the boot media
Step B
Once boot sequence established, initial boot loader copied to internal RAM
After this execution shifts to code loaded into RAM
Bootloader
Program that is executed before operating system starts to function
There are 2 stages
Initial Program Load (IPL)
Detect and setup external RAM
Second Program Load (SPL)
Once external RAM available, SPL copied to RAM and execution transferred
Provides access to other boot modes such as fastboot and recovery
Initiates hardware components such as console, display, keyboard, file system, virtual memory, and other features
Step C: SPL look for Linux kernel
Load this from boot media and copy to RAM
Once bootloader done, transfer execution to kernel
Linux Kernel
Heart of Android system and responsible for process management, memory management and enforcing security on device
After kernel loaded, mounts rootfs and provides access to system and user data
Consists of 2 steps
Step A
When memory management units and caches have been initialized, system can use virtual memory and launch user space processes
Step B
Kernel will look in rootfs for init process and launch it as initiate user space process
init process
First process and root of all other processes
Consists of 2 steps
Step A
Init process will look for script init.rc
Describes the system services, filesystem and other parameters that need to setup
/system/core/init
/system/core/rootdir/init.rc
Step B
init process will parse the init script and launch the system service process
Android logo appear on device screen
Zygote and Dalvik
first init process created after device boots
initialises DVM and create multiple instances to support each Android process
DVM is machine that executes Android applications written in Java
Zygote registers a server socket for zygote connections and preloads certain classes and resources
System Server
Core services get started in this process

Android Forensics Flowchart
What is Mobile Forensics?
Branch of digital forensics that deals with extracting, recovering and analyzing digital evidence or data from mobile device under forensically sound conditions
Deals with accessing the data stored on device, which includes SMS, contacts, call records, photos, videos, documents, application files, history and so on
Important to recover deleted data using various forensic techniques
If evidence has to be admitted in court of law, it's important to work only on image file and not on original device itself
Evidence integrity checks ensure that evidence has not been tampered with from time it was collected
What are the challenges in Mobile Forensics?
Prevent data alteration on device
Switching on device might also change certain state variable present on device
Background processes always run, sudden transition from one state to another can result in loss or modification of data
High possibility of remotely change or delete the contents of the device
Wide range of Operating systems and Device models
Mobile devices differ in OS versions, hardware and various other features
Based on manufacturer, approach to acquire forensic artifacts changes
Within same OS the data storage options and file structures also change, making it even more difficult
Inherent security features
Manufacturers moving towards implementing robust security controls on device, which complicates process of gaining access to data
Difficult for examiners to break these encryptions using techniques such as brute force
Legal issues
Mobile devices can be involved in crimes that span across globe and can cross geographical boundaries
Android Permission Model
Permission model for individual apps
Applications must declare which permissions they require
This model allows user to use app without granting all permissions requested by application
Permissions are stored in AndroidManifest.xml
Categories of permissions
Normal
Low level risk permissions
Do not pose risk to other applications, system or user
Permission automatically granted to user without asking for user approval during installation
Dangerous
These permissions can cause harm to system and other application
User approval is necessary during installation
Signature
These are automatically granted to requesting app if app is signed by same certificate as one that declared/created the permission
This level designed to allow apps that are part of a suite
Signature/System
Permission that system grants only to applications that are in Android system image or that are signed with same certificate as application that declared the permissions
Security Enhanced Linux (SELinux)
Started with Android 4.3
Based on DAC(Discretionary Access Control)
Applications can ask for permissions and users can grant or deny those permissions
Android uses SELinux to enforce MAC (Mandatory Access Control) to ensure applications work in isolated environments
Operate in 2 global modes
Permissive Mode
permission denials are logged but not enforced
Enforcing Mode
Denials are both logged and enforced
Secure Inter-Process Communication
In order to organise data and signals between processes, Inter Process Communication (IPC) framework is needed
In Android, this is achieved by Binder Framework
Provides capabilities required to organise all types of communication between various processes
It is possible to perform variety of actions such as invoking methods on remote objects as if they we local, synchronous and asynchronous method invocation, and sending file descriptors across processes
